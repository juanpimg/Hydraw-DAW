Getting started

#

Dependencies

GCC 8+ / Clang 7+ CMake 3.16+ libgtk-3-dev libwebkit2gtk-4.1-dev libasound2-dev

miniaudio, webview v0.12.0, and nlohmann/json v3.11.3 are fetched automatically via CMake's FetchContent.

Build & run

Three commands from a clean checkout:

$ git clone git@github.com:juanpimg/Hydraw-DAW.git
$ chmod +x scripts/install_deps.sh && ./scripts/install_deps.sh
$ cmake -B build && cmake --build build -j$(nproc)
$ ./build/hydraw_daw
Release build: without an explicit -DCMAKE_BUILD_TYPE, CMake defaults to Release (flags: -O3 -march=native -ffast-math -flto). For debug: cmake -B build -DCMAKE_BUILD_TYPE=Debug.

Project targets

TargetTypeDescriptionDependencies
hydraw_coreSTATICAudio engine + persistence + CLAP host. Zero WebKit/GTK.pthread, dl, asound, nlohmann_json
hydraw_dawEXECUTABLEWebKit/GTK bridge. Links core + webview + X11.hydraw_core, webview::core, GL, X11, asound, pthread, dl, rt

Both targets are defined in a single CMakeLists.txt at project root (~100 lines).

System architecture

#

Hydraw is split into two build targets: hydraw_core — a pure static library with zero UI dependencies — and hydraw_daw — the WebKit/GTK/X11 bridge executable. This separation means the audio engine can be reused by any front-end (headless tests, mobile companion, macOS, etc.) without pulling in WebKit or GTK.

Data flow

// Audio callback (256-frame blocks, 48 kHz, ~5.3 ms):
  miniaudio → AudioEngine::process()
               ├─ Read primary clip (atomic AudioBuffer*)
               ├─ Read ClipsSnapshot (immutable, lock-free)
               ├─ Apply linear fades (gain envelope)
               ├─ Process per-track PluginChain snapshot
               │    └─ clap_plugin->process() for each active plugin
               ├─ Sum sends → aux buses → master bus
               ├─ Write peaks (1 per 256 frames)
               └─ Write to output buffer

// UI update thread (30 Hz):
  buildJsonTelemetry()
    └─ eval("JSON.parse({...})")
         └─ JS draw cycle: waveforms, meters, positions

// Main thread (user actions → bindings):
  cn("nativePlay", "")     → AudioEngine::play()
  cn("nativeSetVolume",  ) → atomic store
  cn("nativeLoadWav", ...) → loadWavToTrack() + publish snapshot

Multi-clip model

Each track has a primary clip stored directly on the Track struct (std::atomic<AudioBuffer*>) for lock-free audio thread access. Additional clips live in a std::vector<Clip> protected by a mutex, published atomically to the audio thread as an immutable ClipsSnapshot* (deep-copied, deferred-deleted).

When the primary ↔ extra transition happens (add, move, remove, rebuildPrimaryFromClips), fade values (primaryFadeIn/Out ↔ Clip.fadeIn/Out) are transferred atomically.

The three pure interfaces

Defined in src/Audio/ and injected via constructor:

interface

IAudioObserver

Event bus: core → bridge. onTelemetry(), onProjectLoaded(), onExportComplete(), onLog() + optional hooks.

interface

ILogSink

Thread-safe logging. log(Level, msg) — lock-free, no-alloc, callable from any thread including audio.

interface

IHostExtensions

CLAP host extension provider. getExtension(id) → const void*.

Note: IAudioObserver is defined but not yet fully adopted — the existing 30 Hz polling loop in main.cpp still drives telemetry. Phase 7 of the migration plan will move to push-based telemetry.

Threading model

# 
┌────────────────────┐    ┌────────────────────┐    ┌──────────────────────┐
│ Main thread        │    │ Audio thread     │    │ UI update thread    │
│ (GTK + WebKit)     │    │ (miniaudio cb)     │    │ (30 Hz timer)        │
├────────────────────┤    ├────────────────────┤    ├──────────────────────┤
│ plugin->init       │    │ Track::audio       │    │ buildJsonTelemetry() │
│ plugin->activate   │    │  (atomic read)     │    │ eval("JSON.parse")   │
│ plugin->gui.*      │    │ ClipsSnapshot*      │    │                      │
│ native bindings    │    │  (immutable copy)   │    │                      │
│ pushUndo/pullSnap  │    │ PluginChain::process│    │                      │
│ ProjectSerializer  │    │  (try_lock never)   │    │                      │
│ loadWavToTrack()   │    │ DelayLine (PDC)     │    │                      │
└────────┬───────────┘    └────────┬───────────┘    └──────────┬───────────┘
         │                        │                          │
         │ clap_host_{log,gui_closed}                          │
         │ g_main_context_invoke ◄─dispatch ──────────────────│
         └────────────────────────┼───────────────────────────┘
                                  │
         ┌────────────────────────┴───────────────────────────┐
         │  Lock-free communication via std::atomic:           │
         │  · TransportState (playing, playhead, totalFrames)  │
         │  · Per-track: volume, pan, mute, solo, armed,       │
         │    peakL/R, frozen, audio (AudioBuffer*), sends     │
         │  · Master/Aux: volume, peakL/R                      │
         │  · ClipsSnapshot* (deferred-deleted on main thread) │
         │  · PluginChain Snapshot* (deferred-deleted)          │
         │  · AutomationSnapshot*                              │
         └─────────────────────────────────────────────────────┘
ActionThreadMechanism
Audio processingaudio (miniaudio cb)reads atomics, never locks
Plugin lifecyclemaindlopen, init, activate, destroy
Plugin GUImainX11 window, clap_host_gui dispatch via g_main_context_invoke
CLAP host callbacksplugin's threadIHostExtensions static methods; GTK work redirected to main thread

Project save/loadmainserialize to string; atomic write (temp file + rename)
Export audiodetached workerrenderOffline() → WavWriter → onExportDone eval
Design principle: The audio callback is the highest-priority thread in the process and must never block. Hydraw enforces this with a strict no-lock policy in the audio path — every cross-thread data structure uses std::atomic with relaxed ordering where possible, and mutable state is published via atomic snapshot pointers (immutable once published, deferred-deleted on the main thread).

Snapshot & deferred-delete pattern

Used for: clips, plugin chains, automation data, and MIDI notes.

CLAP hosting

#

Hydraw implements the CLAP (CLever Audio Plugin) standard via dlopen in src/Plugin/. The host provides a lock-free snapshot model for the audio thread, real parameter flush, thread-check extension, and full lifecycle management.

Implemented extensions

audio-ports✓Multi-port query (up to 4 ports, shared buffer)
state✓StringIStream/StringOStream adapters, base64 persistence
gui✓X11 embedding via clap_host_gui; dispatch to main thread
thread-check✓clap_host_is_main_thread / is_audio_thread
timer-support✓CLAP timers via g_timeout_add
log✓Redirect to ILogSink with severity mapping
params✓Real clap_plugin_params_t::flush() via request_flush queue
render✓Offline render mode for freeze/export
note-ports✓MIDI note events via clap_input_events_t
latency✓PDC via DelayLine per track + master

Lifecycle contract

// Loading:
  clap_host_t host = { host_data, clap_host_get_extension, ... };
  ClapHost::loadLibrary(path)     → // dlopen + refcount
  createPlugin(desc_idx)           → // factory: entry->init + get_extension
  plugin->init(host)               → // configured with sample_rate, block_size
  plugin->activate(48000, 256)     → // allocate processing resources
  plugin->start_processing()       → // signal that audio thread will call process()

// Processing (audio thread):
  Snapshot* snap = m_snapshot.load();          // lock-free read
  clap_process_t process = { ... };
  plugin->process(process) → CLAP_PROCESS_*;

// Cleanup (main thread):
  plugin->stop_processing()
  plugin->deactivate()
  plugin->destroy()
  ClapHost::unloadLibrary(path) → // dlclose
Multi-port limitation: All ports share the same buffer (kMaxPorts = 4). Multi-output plugins produce incorrect output on secondary port bands. This is a known gap (see Limitations).

Architecture decisions

#

All 13 ADRs from DECISIONS.md, chronologically ordered. These record the rationale behind every significant architectural choice.

#DecisionContext
ADR-000ADR formatStandardized lightweight ADR template with Context/Decision/Consequences
ADR-001Modular architectureSeparate audio engine + UI thread with atomic shared state
ADR-002miniaudio over RtAudioSingle-header, no deps, MIT license, ALSA/Pulse/JACK
ADR-003ImGui + GLFW (superseded)Original UI stack, replaced by ADR-009 (WebView)
ADR-004Two-thread concurrencyAudio thread never blocks; atomics for all shared state
ADR-0054-region layoutTimeline + File browser + FX Chain + Mixer
ADR-006CLAP over VST3Open standard, MIT license, no COM, simple dlopen on Linux
ADR-007ma_decoder + atomic playheadWAV loading via miniaudio, lock-free transport state
ADR-008Dynamic tracks + master busMAX_TRACKS=16, atomic track count, MasterBus struct
ADR-009ImGui/GLFW → WebViewReplaced with WebKit2GTK + HTML5+Tailwind+vanilla JS
ADR-010Dark theme (Ableton-inspired)High-contrast carbon + blue palette, WCAG AA, CSS custom properties
ADR-011Multi-clip expansionSnapshot model for clips, automation, MIDI, buses, freeze, undo
ADR-012JSON telemetry over array-evalJSON.parse() avoids WebKitGTK JSC array bugs; generation counter
Why CLAP? CLAP is the only modern plugin standard designed for Linux from day one. It requires no COM, no SDK license, and plugins are just .so files loaded via dlopen. VST3 support could be added later as a bridge layer — the architecture is ready for it.

  • miniaudio
  • PCM buffers
